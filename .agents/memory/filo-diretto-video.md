The video promotes an HR SaaS ("Filo Diretto" wordmark) whose real marketing brand is **HR2Wallet** (site: hr2wallet.netlify.app). Italian-only audience.

**Brand to match (editorial style):** black #0A0A0A bg, cream #F4EDE2 text, electric purple #8B5CF6 / #7C3AED accent — NO teal. Display font **Anton** (big condensed headlines), body **Inter**. Magazine/editorial feel, restrained motion. Real concrete copy: "HR parla. Tutti leggono.", 95%+ lettura, zero app, push sulla lock screen, automazioni (compleanni/anniversari/reminder/quiz/survey), pass also works as QR/NFC badge.
**Why:** the user rejected the first cut (teal + glowy "filo" thread + geometric sans) as off-brand and "grafica scarsa" with overlapping text; they want it aligned to the live site.
**Naming decision:** keep the wordmark **"Filo Diretto"** (user chose this over "HR2Wallet") even though the site says HR2Wallet.

**Hard structural constraint — audio sync:** `composite_audio.mp3` is pre-mixed against `SCENE_DURATIONS` (keys/order/values) in VideoTemplate.tsx. Do NOT change scene keys/order/durations without re-mixing the composite (per video-js audio.md: adelay per scene-start+300ms, music vol 0.20, -t 90). To make pacing "faster" without desync, speed up the per-scene phase timers / animation easings, not the durations.
**How to apply:** any future edit that touches scene timing must either keep SCENE_DURATIONS identical or regenerate composite_audio.mp3 in lockstep. Scenes use grid/flex with `pt-28 pb-28` safe padding to clear the persistent top/bottom editorial chrome — keep that to avoid text overlap.

**Lesson — delegate video-js work to the DESIGN subagent (don't hand-build):** the first two cuts were built directly by the main agent with only text + CSS shapes; the user rejected them as "grafica scarsa", "manca impaginazione", and not "umano". The fix that landed was delegating the iteration to the DESIGN subagent, which generated/downloaded real stock photography of people and produced editorial layouts.
**Why:** the DESIGN subagent has image-generation / stock-photo / web-search tools the main agent lacks; a "human", produced feel for this product *requires* real imagery of people (hands holding a phone with a lock-screen, work environments), not shapes+type. Hand-built video-js output reads as a slideshow.
**How to apply:** for any video-js build OR iteration, delegate to the DESIGN subagent per the video-js skill; pass user feedback verbatim + the hard constraints (locked SCENE_DURATIONS, name, don't-touch audio/hooks). Main agent still owns audio (composite mix) and workflow restart/verification afterwards.

**Content preference — avoid the acronym "HR" in on-screen Italian copy:** the user finds "HR" awkward to pronounce in Italian. Use "Comunicazione Interna", "Pannello Personale", or "L'azienda" instead. The product wordmark stays "Filo Diretto".
**Feature beats the user wants emphasized (employee-facing):** badge (the wallet pass doubles as NFC/QR company badge, works offline), engagement (real, measured), reward (premi erogati), and especially **100% legal/privacy compliance** ("a norma di privacy") — the legal-reassurance beat matters most when addressing employees, so give it clear visual weight, not a footnote.

**Spoken copy lives in the recorded voiceover, not the on-screen text.** Changing scene JSX does NOT change what the narrator says. The Italian VO is per-scene (public/audio/vo_<scene>.mp3) in voice "Paolo" (voiceId `mcMi8FJDhg35bMpWHv2R`, model eleven_multilingual_v2, settings stability 0.5 / similarity 0.75 / style 0.3 / speaker_boost). **Generate VO via the media-generation `textToSpeech({text, voiceId, outputPath, voiceSettings})` callback in code_execution — NOT an ElevenLabs connector** (`listConnections('elevenlabs')` returns 0; the managed TTS service is what's used). To change a spoken word, regenerate ONLY that vo_<scene>.mp3 with the SAME voiceId (keep duration under the scene window minus ~0.4s tail), then rebuild public/audio/composite_audio.mp3.
**Composite recipe (CURRENT = 8 scenes):** order/durations in SCENE_DURATIONS are context10000 / problem12000 / intro10000 / dashboard16000 / wallet14000 / push14000 / analytics12000 / outro10000 → total 98000ms. ffmpeg: `-stream_loop -1 -i bg_music.mp3` (music is only 90s, MUST loop to cover 98s) at volume 0.20; each VO volume 1.0 with adelay=sceneStart+300ms at starts 0/10000/22000/32000/48000/62000/76000/88000 ms; amix inputs=9 duration=longest normalize=0; clamp `-t 98`. VideoTemplate computes SCENE_START_SEC from SCENE_DURATIONS order automatically, so durations + composite must move in lockstep.

**Scene safe-area: every scene must clear the fixed editorial chrome.** VideoTemplate renders a top bar (~48px) and a bottom bar with the progress dots/label (~57px) as fixed overlays. Scene content sits inside `py-20`-ish padding; do NOT add an extra `pt-10` on the text column on top of a `justify-center` container — it biases content downward and makes photos/text sit "troppo attaccato sotto" (too close to the bottom bar). Keep content vertically centered, and for full-height photo columns give a slightly larger bottom padding (e.g. pb-28) than top. Floating cards over a photo panel (Scene4 pass/badge) must keep a generous right margin (~16% of the half-width container) so they don't get clipped by the panel's overflow-hidden / read as overflowing the frame.

**Avoid overflow-prone Tailwind on scene visuals.** When enlarging a scene's main visual (dashboard mockup, pass card), do NOT use `h-[110%]` + `transform scale-110` + negative margins (`-mr-12`) to make it "bigger" — that bleeds past the px-24/py-20 safe padding into the fixed top/bottom chrome and frame edges (the user's recurring complaint). Instead enlarge with explicit sizes (`h-[500px]`, larger `w-[..px]`) kept inside `h-full`/the safe padding. For floating accent chips placed over a card (e.g. "Convenzioni locali" pill on the pass), check it doesn't land on the card's QR/content — keep the card slightly higher (`top-[42%]`) and the chip low/left (`bottom-[6%] left-[8%]`) so they don't overlap.

**Layout must hold at any resolution, not just 1280×720.** The video is authored/verified at 1280 but the canvas iframe and the recording harness render the live DOM at larger sizes (e.g. 1920) — overflow only appears there. Two durable rules learned from the "errori di impaginazione at 1920" bug:
- **Never size scene headlines with raw viewport units (`text-[Nvw]`)** when body text/spacing are fixed px — the headline balloons ~50% at 1920 and shoves stacked content into the fixed bottom chrome. Always `clamp(minPx, Nvw, maxPx)` with the cap near the authored-1280 px value so 1280 is unchanged and 1920 can't overflow.
- **Never put two `position` utilities on one element** (e.g. `absolute … relative`). Class order in the string is irrelevant; Tailwind's CSS defines `relative` after `absolute` so `relative` wins, silently keeping a meant-to-be-absolute panel in flow where it pushes siblings off-frame. Grep `rg 'absolute.*relative"'` after layout edits.
- **Reliable way to screenshot one mid-timeline scene fully-revealed at a target size:** the plain `screenshot` tool only shows the first scene at t≈0 and races opacity transitions; `runTest` drifts and often returns no paths. Deterministic method = temporarily reorder SCENE_DURATIONS so the scene is first AND force-reveal it (max `useState` phase + set every motion `initial` to its revealed value — useState alone isn't enough), then revert all temp edits. **Why:** verified the clamp fix alone was insufficient and only the forced-reveal shot exposed the real absolute/relative overflow.

**Badge & notifications are shown inside a reusable PhoneFrame device mockup.** There is a shared `src/components/video/PhoneFrame.tsx` (portrait device bezel + dynamic island; `children` render on the screen). Scene4 renders the Wallet pass on the phone screen; Scene5 renders the lock-screen notifications on it. Keep this device-mockup approach for any future "su un telefono" requests rather than floating bare cards. The phone is ~270x550 and sits in the right 50% panel — that height nearly fills the safe area, so do NOT make the phone taller or it will hit the fixed top/bottom chrome. Floating accent chips (e.g. "Convenzioni locali") should overlap the phone's lower edge, never the on-screen pass/QR content.
